PhosphorTEX — Decoder Test Results

Accuracy testing of the PhosphorTEX NAVTEX decoder using the CLI test harness (NavTexDecode). The CLI uses identical DSP code to the iOS app via symlinks — same source files, same binary logic.

Methodology

  • All tests run via the NavTexDecode CLI tool, which shares DSP source files with the iOS app via symlinks (single source of truth).
  • Each synthetic scenario is generated from the CCIR-476 wire format specification: alpha / rep interleaving with a 2-character offset and a 2×[code_rep, code_alpha] preamble that triggers the receiver's rep-on-alpha resync.
  • FEC statistics are reported per the winning candidate of the 7-way phase disambiguator: DX = alpha copy 4:3-valid, RX = fell back to rep copy, ERRS = both invalid, MISM = both valid but disagreed.
  • CARDS output = framed ZCZC … NNNN message; TAPE output = raw merged character stream.

Decoder Settings

  • Sample rate: 8 kHz (resampled from source if needed)
  • FFT: 512-point, Hann window, ~15.6 Hz/bin
  • Tone detection: I/Q quadrature demodulation, biquad LPF (102 Hz cutoff at 170 Hz shift)
  • Envelope: asymmetric decay-average (attack = symbollen/4, decay = symbollen×16)
  • Bit decision: mark / space magnitude compare with hysteretic squelch (open > 0.002, close > 0.001)
  • SitorDecoder: synchronous 7-bit framer, DPLL α = 0.05
  • FECMerger: 5-character DX/RX pair-merge delay per CCIR-476 spec
  • PhaseDisambiguator: 7 parallel SitorDecoders with lockThreshold = 8..14, scored by (mismatched + 2×bothInvalid) / contentCharsIn, decisive commit below 0.05
  • AFC: parabolic sub-bin peak interpolation (3-point fit), centre tracking with snap on acquisition and ±3 Hz/frame clamp in steady-state, 1 Hz retune threshold, shift snapped to 170 Hz exact per the CCIR-476 specification
  • NAVTEX protocol parameters (fixed): 100 Bd, 170 Hz shift, CCIR-476 alphabet (4-marks + 3-spaces validity test)

Specification Basis

PhosphorTEX’s DSP is built from the published CCIR Recommendation 476 / 625 specification for SITOR mode B, the IMO NAVTEX Manual (IMO publication 951E) for NAVAREA station identifiers and slot allocations, and ITU-R Recommendation M.540 for the 170 Hz shift and 100 Bd rate. Where we import test-audio files for cross-checking, their provenance is noted in the corresponding row of the test table below.

Notes on Specific Files

navtex_fec_stress.wav — Bit-Flipped REP Copies

Every REP copy on the wire has its least-significant bit flipped, stressing the FEC fallback machinery. Our merger correctly prefers the uncorrupted alpha (DX) copy whenever it’s 4:3-valid, so body characters still decode. The elevated 49% mismatch rate is the signature of the corruption: alpha and rep are both 4:3-valid in many cases (a corrupted rep that happens to land on another of the 35 valid codes) but disagree. CARDS is empty because enough register-shift codes land at corrupted positions mid-body that LTRS/FIGS state drifts and the framer loses the ZCZC / NNNN match by the end. This is a worst-case synthetic — no real-world channel bit-flips every REP. Real FEC falls back on a few percent of REPs, which our merger handles cleanly.

navtex_phasing_only.wav — Idle-Channel Silence

The wire carries only α/β phasing symbols — what a NAVTEX transmitter emits between messages. All seven phase candidates see valid codes but every emission is a phasing-filler byte, filtered out by the merger. The disambiguator correctly declines to commit (winner = none) and the framer stays .idle. No noise is streamed to TAPE or CARDS — the correct behaviour when no real message is on-air.

navtex_long_nnnn_missing.wav — Dropped Terminator

Simulates a station that skipped the NNNN sentinel. The framer enters .body state on ZCZC detection, accumulates characters, and would normally close when NNNN arrives. Because it never does, the end-of-file flush path fires (same code path as the app’s STOP button) and the message is emitted with truncated: true. The body is decoded cleanly up to the point of EOF.

Known Limitations

Real-world OTA recordings at non-canonical carriers (e.g. the fventuri examples at ~1063 Hz centre) currently acquire AFC lock and hold it without retune thrash, but the Goertzel-based tone detectors do not emit bits at that frequency — under investigation. Possible causes: squelch threshold tuned for a stronger signal level, or the downsampled 8 kHz file lacks the spectral content the 102 Hz LPF expects. The decoder remains verified on canonical ~1615 Hz mark NAVTEX signals.

Summary

  • Clean signal: 178 characters, zero errors, full envelope framed, message card rendered correctly.
  • Missing NNNN: full body decoded, correctly flagged [TRUNC] via EOF flush.
  • Idle channel: correctly silent — no phantom messages, no noise streamed.
  • Phase ambiguity: the 7-way disambiguator picks the correct character-boundary phase decisively (0.0% vs 10%+ on other phases) and replays its buffered history into the framer so the ZCZC header is preserved.
  • FEC: DX preferred on valid alpha, RX fallback on alpha failure, ERR on both invalid — CCIR-476 spec-conformant behaviour verified.
  • Open work: the Goertzel tone detectors need investigation to decode real-world OTA samples at non-canonical carrier frequencies.
